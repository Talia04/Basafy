## Critical Security Issues

### 1. **Plaintext Refresh Token Storage** 🔴 CRITICAL
- **File:** `basafy-backend/supabase/migrations/20241002120000_add_gmail_tables.sql`
- **Line:** 10
- **Issue:** The refresh token is stored in plaintext in the database. This is a critical security issue as refresh tokens provide long-lived access to user Gmail accounts.
- **Recommendation:** The refresh_token column should be encrypted at rest using PostgreSQL's pgcrypto extension or application-level encryption before storing.
- **Suggested Change:** 
  ```sql
  refresh_token bytea not null,
  ```

### 2. **Plaintext Access Token Storage** 🔴 CRITICAL
- **File:** `basafy-backend/supabase/migrations/20241002120000_add_gmail_tables.sql`
- **Line:** 11
- **Issue:** The access token is also stored in plaintext. While access tokens are short-lived, they should still be encrypted.
- **Consideration:** Consider whether storing the access token is necessary at all - it can be regenerated from the refresh token when needed.
- **Suggested Change:**
  ```sql
  access_token bytea,
  ```

### 3. **Insecure OAuth Token Extraction** 🔴 CRITICAL
- **File:** `basafy-mobile/basafy-rn-expo/src/screens/Onboarding/GmailImportOnboarding.tsx`
- **Line:** 314
- **Issue:** The provider refresh token and auth code are extracted from the URL hash fragment, which is inherently insecure as URL fragments can be logged by browser history, analytics tools, and intermediate proxies. This is a known OAuth security anti-pattern.
- **Recommendation:** Consider using the authorization code flow exclusively with PKCE instead of relying on tokens in the URL fragment.

### 4. **Exposed Environment Variables in Logs** 🔴 HIGH
- **File:** `basafy-backend/supabase/functions/gmail-sync-user/index.ts`
- **Line:** 134
- **Issue:** The environment variables GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET are logged or potentially exposed in error messages.
- **Recommendation:** Avoid including sensitive credentials in logs or user-facing error messages. Instead, use generic error messages like "Service configuration error" without revealing which specific credential is missing.

- **File:** `basafy-backend/supabase/functions/gmail-sync-user/index.ts`
- **Line:** 336
- **Issue:** Messages are fetched sequentially in a for loop, which will be slow for large batches. This N+1 query pattern could significantly delay the sync operation.
- **Recommendation:** Consider using Promise.all or Promise.allSettled to fetch multiple messages concurrently, with appropriate rate limiting to respect Gmail API quotas.

### 7. **No Connection Pooling** 🟡 MEDIUM

- **Line:** 156
- **Issue:** The function creates a new Supabase client instance on every invocation but never explicitly closes connections. While Deno may handle cleanup, this could lead to potential connection leaks, especially under high load.
- **Recommendation:** Consider implementing proper connection pooling or reusing client instances.
